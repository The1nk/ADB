# M11 — OCR — Design

**Status:** Approved
**Date:** 2026-06-04
**Milestone:** M11 (post-V1 roadmap; OCR is net-new, not in V1)

---

## 1. Overview

Add OCR (optical character recognition) so bots can read and act on on-screen text. Four actions — **Read Text**, **Find Text**, **Wait for Text**, **Assert Text Absent** — each for both **Screen** (Win32 window capture) and **Android** (framebuffer) targets. Browser is excluded (it has native DOM `Get Text`). The design mirrors the M10a image-matching family: capture → ROI crop → analyze → write results, with the template matcher swapped for an OCR engine and the match-variable contract reused so Find Text feeds Tap/Click exactly like Find Image.

## 2. Engine

Tesseract via the `charlesw/Tesseract` NuGet, shipping **always-on** (the M5 "OpenCvSharp ships always-on" precedent), with a bundled `eng.traineddata` so OCR works with zero user setup. The NuGet supplies the native `tesseract`/`leptonica` binaries; `eng.traineddata` is committed to the repo and copied to each executable's output `tessdata/` folder. `TesseractOcrEngine` resolves `tessdata` relative to `AppContext.BaseDirectory`.

## 3. Architecture

### 3.1 Engine abstraction
```
namespace AdbCore.Ocr;

public interface IOcrEngine
{
    OcrResult Recognize(System.Drawing.Bitmap image);
}

public readonly record struct OcrWord(string Text, System.Drawing.Rectangle Bounds, double Confidence); // Confidence 0–1
public sealed record OcrResult(string Text, IReadOnlyList<OcrWord> Words);
```
- `TesseractOcrEngine` is the concrete, live-verified implementation. Recognized word confidence (Tesseract 0–100) is normalized to 0–1.

### 3.2 Shared `OcrCore` (reuses `TemplateMatchCore`)
`OcrCore` is thin and depends on the existing `TemplateMatchCore` for region + variable handling:
- `RecognizeRegion(Bitmap, config, IOcrEngine) → OcrResult`: crops to the configured ROI via `TemplateMatchCore.ResolveRegion` (+ the same clone-crop), runs `Recognize`, and offsets every word's `Bounds` back to full-frame coordinates by the ROI origin.
- `FindWord(OcrResult, string target, double minConfidence) → MatchResult?`: case-insensitive; returns the first word whose `Text` **contains** the (trimmed, lowercased) target and whose confidence ≥ `minConfidence`, as a `MatchResult(Bounds.X, Bounds.Y, Bounds.Width, Bounds.Height, Confidence)`.
- Find Text writes its result through the **existing** `TemplateMatchCore.WriteMatchVariables(variables, MatchResult, prefix, random)`, so the output contract (`matchLeft/Top/Right/Bottom`, `matchCenterX/Y`, `matchRandX/Y`, `matchConfidence`) is identical to Find Image.

### 3.3 Action bases (mirror M10a)
- `ScreenOcrActionBase` — constructor-injects `IWindowCapture` + `IOcrEngine`; resolves the target HWND, captures the client area, delegates to `OcrCore`. Category "Screen".
- `AndroidOcrActionBase` — bound `IAndroidDevice` (handle) + injected `IOcrEngine`; captures the framebuffer (decode PNG → Bitmap), delegates to `OcrCore`. Category "Android". No `captureMethod` field.

## 4. Actions

Eight actions = 4 behaviors × {Screen, Android}. Type keys: `screen.readText` / `android.readText`, `screen.findText` / `android.findText`, `screen.waitForText` / `android.waitForText`, `screen.assertTextAbsent` / `android.assertTextAbsent`.

| Action | Config (besides shared ROI) | Behavior |
|---|---|---|
| **Read Text** | `resultVar` (default `"text"`) | OCRs the region; writes the full recognized text (trimmed) to `resultVar`. Always succeeds (out port). |
| **Find Text** | `text` (target), `resultVar` (default `"match"`), `minConfidence` (default `0`) | Finds the first word containing `text`; writes the match variables under `resultVar`. Not found → **Fail** (drives retry / onFailure). |
| **Wait for Text** | `text`, `resultVar`, `minConfidence`, `timeoutMs` (5000), `pollIntervalMs` (250) | Polls Find Text until the word appears or the timeout elapses; timeout → Fail. |
| **Assert Text Absent** | `text`, `minConfidence` | Succeeds only when no word contains `text`; present → **Fail** (so with retry it becomes "wait until gone"). |

Shared ROI fields are `TemplateMatchCore.RegionFields()` (regionX/Y/Width/Height). `${var}` interpolation applies to all string/number config (engine-side, unchanged). Find/Wait/AssertAbsent `SupportsRetry = true`; Read Text `SupportsRetry = false`.

## 5. Find Text matching (v1)

Case-insensitive, **word-level substring**: the first OCR word whose lowercased text contains the lowercased, trimmed target (and meets `minConfidence`) is the match. Its bounding box (offset to full-frame) and OCR confidence populate the match variables. **Deliberate v1 limitation:** targets that span multiple OCR words (phrases) are not matched as a unit — single words / substrings (typical for buttons and labels) are the supported case. Phrase-spanning matching is a possible later extension.

## 6. Slicing (mirrors M10)

- **M11a** — OCR engine + core + the 4 **Screen** actions: `IOcrEngine`/`OcrResult`/`OcrWord`, `TesseractOcrEngine` (+ tessdata bundling), `OcrCore`, `ScreenOcrActionBase`, the 4 `screen.*` actions, registration. AdbCore-only.
- **M11b** — the 4 **Android** actions: `AndroidOcrActionBase` + the 4 `android.*` actions, reusing the engine/core, registration.

Both reuse the same engine + core; M11b adds only the Android capture path.

## 7. Testing

- **Unit (AdbCore.Tests):** the OCR actions + `OcrCore` are tested against a **fake `IOcrEngine`** returning canned `OcrResult`s (text + word boxes + confidences). Cover: Read Text writes the variable; Find Text match (variables + box offset by ROI) / no-match Fail / minConfidence filtering; Wait for Text success + timeout; Assert Text Absent present-Fail / absent-Ok; ROI crop offset; case-insensitivity. The shared `TemplateMatchCore` reuse keeps the Find Text output contract identical to Find Image (already covered by `TemplateMatchCoreTests`).
- **Live-verified (user):** `TesseractOcrEngine` reading real text from a real Screen capture (M11a) and a real Android framebuffer (M11b) — including the tessdata bundling working from the deployed output. The actions themselves are unit-covered; only the concrete engine + bundling need hands-on verification.
- **Merge handling:** both slices need the live OCR check, so neither self-merges; each is opened as a PR and verified by the user.

## 8. Out of scope

- Non-English languages (only `eng.traineddata` bundled; the engine could accept others later).
- Phrase/multi-word Find Text matching (v1 is word-level).
- Browser OCR (DOM `Get Text` already exists).
- Any cloud OCR (V1 non-goal).

## 9. Deployment note

`eng.traineddata` is a ~15–30 MB binary committed to the repo under a shared location (e.g. `assets/tessdata/eng.traineddata`) and copied to the output `tessdata/` of each executable that runs OCR (BotRunner; BotBuilder only if it ever OCRs in-process — currently it does not, so BotRunner is the primary). The `Tesseract` NuGet's native binaries are restored automatically. `TesseractOcrEngine` constructs with the resolved `tessdata` directory and language `"eng"`.
