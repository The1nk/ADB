# ADB: A Damn Bot — Software Design Document

**Status:** Approved  
**Last Updated:** 2026-06-01

---

## 1. Overview

ADB: A Damn Bot is a Windows desktop automation tool — an approachable, visual alternative to AutoHotkey with a focus on mobile (Android) automation. Users build **bots** using a drag-and-drop node-graph editor, then run them independently via a separate console executable.

The guiding philosophy is **approachability without sacrificing power** — the Bot Builder should feel like a toy you can pick up instantly, while the Bot Runner should be reliable and scriptable enough for power users. If the built-in actions aren't enough, a Lua scripting escape hatch covers the rest.

---

## 2. Goals & Non-Goals

### Goals
- Allow non-technical users to build bots visually via drag-and-drop
- Support a wide range of automation: screen/image recognition, Android (ADB), browser, file/system, and API
- Keep bots as portable, human-readable `.bot` files
- Provide a clean separation between editing and execution concerns
- Run on Windows with C# / .NET

### Non-Goals (for now)
- Cloud sync or remote execution
- Multi-user collaboration
- Cross-platform support (macOS/Linux)
- A code-based bot authoring mode
- A built-in installer — optional dependencies (ADB, Playwright browsers) are user-installed; see README

## 4. Core Library (`AdbCore`)

This library is the foundation of both executables. It owns:

- **Domain models** — the data structures representing a bot
- **Serialization** — reading and writing `.bot` files
- **Action registry** — the catalogue of available action types
- **Execution engine** — the runtime that walks the graph and executes actions

### 4.1 Domain Model

```csharp
// A bot is a named DAG of actions and the connections between them
public class Bot
{
    public Guid Id { get; set; }
    public string Name { get; set; }
    public string Description { get; set; }
    public List<BotTarget> Targets { get; set; }   // named targets, resolved at run start
    public List<BotAction> Actions { get; set; }
    public List<ActionConnection> Connections { get; set; }
    public DateTime CreatedAt { get; set; }
    public DateTime UpdatedAt { get; set; }
}

// A named automation target — window, Android device, or browser context
public class BotTarget
{
    public Guid Id { get; set; }
    public string Name { get; set; }               // user-defined, e.g. "Client 1", "My Phone"
    public BotTargetType Type { get; set; }
    public Dictionary<string, string> Config { get; set; }
    // Window:  { "selector": "process:BlueStacks" }
    // Android: { "selector": "serial:emulator-5554" }
    // Browser: { "selector": "url:https://..." }
}

public enum BotTargetType
{
    Window,         // Win32 HWND / FlaUI — desktop apps or emulators
    AndroidDevice,  // ADB device
    Browser,        // Playwright browser context
}

// A single action node in the graph
public class BotAction
{
    public Guid Id { get; set; }
    public string TypeKey { get; set; }         // e.g. "screen.findImage", "android.tap"
    public string Label { get; set; }           // user-editable display name
    public Guid? TargetId { get; set; }         // which target this action operates on; null = first target
    public Dictionary<string, object> Config { get; set; }  // action-specific settings
    public RetryPolicy? Retry { get; set; }     // optional per-action retry config
    public Position CanvasPosition { get; set; }
}

// Retry configuration — attached per-action for flaky operations
public class RetryPolicy
{
    public int MaxAttempts { get; set; }        // e.g. 3
    public int DelayMs { get; set; }            // delay between attempts
}

// A directed edge between two actions
public class ActionConnection
{
    public Guid Id { get; set; }
    public Guid SourceActionId { get; set; }
    public string SourcePort { get; set; }      // e.g. "onSuccess", "onFailure", "true", "false"
    public Guid TargetActionId { get; set; }
    public string TargetPort { get; set; }      // e.g. "input"
}
```

### 4.2 Variable System

Variables are simple key/value pairs stored in a shared `ExecutionContext` dictionary that flows through the entire run. Values are strings, numbers, or booleans — for complex data (e.g. JSON payloads), the Lua scripting action handles serialization/deserialization internally.

At run start, the engine resolves all declared `BotTarget`s into live handles (HWND, ADB device, Playwright page) and stores them in the context keyed by `TargetId`. Actions pull their handle from context without any per-action lookup overhead.

```csharp
public class ExecutionContext
{
    public Dictionary<string, object> Variables { get; set; } = new();
    public Dictionary<Guid, ResolvedTarget> Targets { get; set; } = new();
    // Actions read and write variables by name via Set Variable / scripting actions
    // Actions retrieve their target handle via context.Targets[action.TargetId ?? defaultTargetId]
}

public class ResolvedTarget
{
    public BotTargetType Type { get; set; }
    public object Handle { get; set; }   // IntPtr (HWND), IAdbClient device, IPage (Playwright)
}
```

### 4.3 Action System

Each action type is defined by implementing `IActionDefinition`. This is the primary extension point.

```csharp
public interface IActionDefinition
{
    string TypeKey { get; }             // unique key, e.g. "screen.findImage"
    string DisplayName { get; }
    string Category { get; }            // e.g. "Screen", "Android", "Control Flow"
    string Description { get; }
    List<PortDefinition> InputPorts { get; }
    List<PortDefinition> OutputPorts { get; }
    List<ConfigField> ConfigFields { get; }   // drives the properties panel UI
    bool SupportsRetry { get; }               // whether retry policy is applicable
}

public interface IActionExecutor
{
    Task<ActionResult> ExecuteAsync(ActionExecutionContext context, CancellationToken ct);
}

public class ActionResult
{
    public bool Success { get; set; }
    public string? ErrorMessage { get; set; }
    public string OutputPort { get; set; }          // which port to follow next
    public Dictionary<string, object> Outputs { get; set; }
}
```

#### Built-in Action Categories

| Category | Backed By | Example Actions |
|---|---|---|
| **Control Flow** | Engine-native | Start, End, Branch (if/else), Loop (count/for-each), Delay, **Run Parallel**, **Join** |
| **Screen** | OpenCvSharp | Find Image, Wait for Image, Screenshot, Assert Image Absent |
| **Input** | Windows APIs | Click, Right Click, Double Click, Type Text, Key Press, Mouse Move |
| **Android** | SharpAdbClient + ADB | Tap, Swipe, Screenshot, Press Back, Launch App, Install APK |
| **Browser** | Playwright | Open URL, Click Element, Type, Wait for Selector, Get Text |
| **Web & API** | HttpClient | HTTP GET, HTTP POST, Parse JSON |
| **Files & System** | .NET BCL | Read File, Write File, Copy/Move File, Run Process |
| **Desktop UI** | FlaUI | Find Element, Click Element, Type Text, Wait for Element |
| **Data** | Engine-native | Set Variable, Log, Comment |
| **Scripting** | NLua | Run Lua Script |

> **Note:** Android (ADB), Browser (Playwright), and Desktop UI (FlaUI) require optional external dependencies. If unavailable, those action categories are disabled with a prompt directing users to the README.

#### Graph Constraints

The bot graph is a **DAG (directed acyclic graph)** — cycles are not permitted. Looping is handled exclusively by the Loop action, which manages iteration internally. This prevents infinite loop bugs by construction and keeps the execution model simple.

#### Parallel Execution

**Run Parallel** and **Join** are a matched pair that enable concurrent execution within a bot — most commonly used to drive multiple game clients simultaneously.

- **Run Parallel** has one input port and N named output branch ports (user-configurable). The engine fans out and executes each branch concurrently as independent `Task`s. Each branch action can target a different `TargetId`, which is the natural model for multi-client automation.
- **Join** waits for all branches to complete before execution continues. It exposes `allSucceeded` and `someFailed` output ports so the user can wire different handling paths after the join.
- **On branch failure** is a configurable property on Run Parallel:

```csharp
public enum ParallelErrorStrategy
{
    HaltAll,       // cancel all running branches immediately on first failure
    WaitThenHalt,  // let running branches finish naturally, then halt
    Continue,      // treat failures as warnings; always proceed to Join
}
```

A typical multi-client bot looks like:

```
Start → Run Parallel ──► Branch 1 (Target: Client 1) ──► Join → End
                     ├──► Branch 2 (Target: Client 2) ──►┤
                     └──► Branch 3 (Target: Client 3) ──►┘
```

### 4.4 Execution Engine

The engine walks the graph starting from action(s) with no incoming connections (entry points). It is async and cancellable throughout.

```csharp
public class BotExecutor
{
    public async Task<ExecutionResult> RunAsync(
        Bot bot,
        ExecutionOptions options,
        IProgress<ExecutionProgress> progress,
        CancellationToken ct);
}
```

**Execution model:**
- At run start, all `BotTarget`s are resolved into live handles and stored in `ExecutionContext`
- Actions execute sequentially along connection paths; each action pulls its target handle from context via its `TargetId` (defaulting to the first target if unset)
- Output port names (`onSuccess`, `onFailure`, `true`, `false`) determine which connection to follow
- **Error handling:** halt on failure by default. Actions with `SupportsRetry = true` (e.g. Find Image) may have a `RetryPolicy` configured in their properties — the engine retries up to `MaxAttempts` times before treating the action as failed
- A wired `onFailure` port on any action provides an alternative to halting — execution continues along that branch instead
- **Run Parallel** fans out branches as concurrent `Task`s; **Join** awaits all of them. Branch failure behaviour is governed by the `ParallelErrorStrategy` configured on the Run Parallel action

### 4.5 Serialization

Bots are saved as `.bot` files (JSON under the hood). The format is human-readable and version-tagged. `.bot` files are registered as a Windows file association opening in BotBuilder.

```json
{
  "version": "1.0",
  "id": "a1b2c3...",
  "name": "Farm Gold",
  "targets": [
    {
      "id": "target-1",
      "name": "Client 1",
      "type": "Window",
      "config": { "selector": "process:BlueStacks" }
    },
    {
      "id": "target-2",
      "name": "My Phone",
      "type": "AndroidDevice",
      "config": { "selector": "serial:emulator-5554" }
    }
  ],
  "actions": [
    {
      "id": "action-1",
      "typeKey": "screen.findImage",
      "label": "Find Attack Button",
      "targetId": "target-1",
      "config": {
        "templatePath": "assets/attack-btn.png",
        "confidence": 0.9
      },
      "retry": { "maxAttempts": 5, "delayMs": 500 },
      "position": { "x": 120, "y": 80 }
    }
  ],
  "connections": [
    {
      "id": "conn-1",
      "sourceActionId": "action-1",
      "sourcePort": "onSuccess",
      "targetActionId": "action-2",
      "targetPort": "input"
    }
  ]
}
```

## 5. Bot Builder (`BotBuilder`)

A WPF application. Its job is to let users open, create, and save `.bot` files using a visual node-graph canvas.

### 5.1 UI Layout

```
┌──────────────────────────────────────────────────────┐
│  Menu Bar (File, Edit, Run, Help)                    │
├──────────────────────────────────────────────────────┤
│  Target Bar  [ + Add Target ] [ Client 1 ▾ ] [ My Phone ▾ ]  │
├───────────┬──────────────────────────┬───────────────┤
│           │                          │               │
│  Action   │       Canvas             │  Properties   │
│  Palette  │   (drag-drop graph)      │  Panel        │
│           │                          │               │
│  [Search] │                          │  (selected    │
│           │                          │   action      │
│  Control  │                          │   config)     │
│  Flow     │                          │               │
│  Screen   │                          │               │
│  Android  │                          │               │
│  ...      │                          │               │
├───────────┴──────────────────────────┴───────────────┤
│  Status Bar                                          │
└──────────────────────────────────────────────────────┘
```

The **Target Bar** sits below the menu and displays all configured targets as labelled chips. Clicking a chip opens the target's settings (name, type, selector). The `+` button adds a new target.

### 5.2 Canvas

The canvas is the central editing surface.

**Navigation:**
- **Pan** — middle-click drag or Space + drag
- **Zoom** — scroll wheel, zoom-to-cursor

**Editing:**
- **Select** — click to select; drag marquee to select multiple
- **Connect** — drag from an output port to an input port
- **Disconnect** — click a connection line, press Delete
- **Delete** — Delete key removes selected actions or connections
- **Undo/Redo** — Ctrl+Z / Ctrl+Y

**Action cards** render as rounded-rect cards with:
- A coloured header bar (colour per category)
- Input ports on the left edge, output ports on the right edge
- The action label and a small category icon
- A small target badge (e.g. "Client 1") when the bot has more than one target, making multi-client bots readable at a glance

Connections render as smooth cubic Bezier curves between ports.

### 5.3 Action Palette

A searchable sidebar listing all registered action types, grouped by category. Actions are dragged from the palette onto the canvas. Double-clicking a palette entry adds it at the canvas centre.

Disabled categories (missing optional dependency) are shown greyed out with a tooltip explaining what to install.

### 5.4 Properties Panel

When an action is selected, the right panel renders its `ConfigFields` as a form.

| Field Type | UI Control |
|---|---|
| String | TextBox |
| Multiline string | TextBox (multi-line) |
| Number | NumericUpDown |
| Boolean | CheckBox |
| Enum / select | ComboBox |
| File path | TextBox + Browse button |
| Image path | TextBox + Browse + Preview thumbnail |

A **Target** dropdown always appears at the top of the panel for any action that operates on a target, listing all configured targets by name. Defaults to the first target if only one is configured.

For actions with `SupportsRetry = true`, a **Retry** section appears at the bottom of the panel with Max Attempts and Delay fields.

### 5.5 Test Run

**Run > Test Run** spawns `BotRunner.exe` with the current `.bot` file path as an argument, and tails its log output in a bottom panel. This is a convenience shortcut — the runner is fully independent.

## 6. Bot Runner (`BotRunner`)

A console application that loads a `.bot` file and executes it via `AdbCore`'s execution engine.

### 6.1 Invocation

Targets are resolved at run start via CLI arguments. Each target declared in the `.bot` file must have a corresponding `--target` argument that provides its selector at runtime.

```
BotRunner.exe --bot "C:\Bots\farm-gold.bot" \
              --target "Client 1=process:BlueStacks" \
              --target "My Phone=serial:emulator-5554" \
              [--log-level debug]
```

Designed to be invoked from:
- BotBuilder (test runs)
- Windows Task Scheduler
- Scripts or batch files
- A `.bot` file double-click (via file association → BotBuilder by default; Runner invocable directly)

### 6.2 Target Resolution in BotBuilder (Test Run)

When launching a test run from BotBuilder, a **target picker dialog** is shown instead of requiring CLI args to be typed manually. The dialog:

- Lists all targets declared in the bot
- For each target, shows a live dropdown of available candidates (running processes, connected ADB devices, open browser contexts)
- After the user confirms their selections, **displays the equivalent CLI command** so they can copy it for use in scripts or Task Scheduler

This bridges the gap between approachable UI and scriptable CLI — users learn the CLI naturally through the tool rather than from docs alone.

### 6.2 Logging & Output

- Logs to `stdout` for scripting compatibility
- Writes a structured log file alongside the `.bot` file (or a configurable path)
- Log format: JSON lines, one entry per action execution
- Exit code `0` = success, non-zero = failure (with error details in log)

## 7. Bot Capture (`BotCapture`)

A lightweight WPF tool for capturing and validating template images used by screen-matching actions (e.g. Find Image, Wait for Image). It can be launched standalone for iterating on a template library, or from BotBuilder to fill a specific image path field.

### 7.1 Invocation Modes

**Standalone** — launched directly by the user, no arguments:
```
BotCapture.exe
```
Stays open across multiple captures. The user names and saves templates to a folder of their choosing. Useful for building out a template library before or during bot authoring.

**Integrated** — launched by BotBuilder when the user clicks the capture button on an image path field:
```
BotCapture.exe --output "C:\Bots\assets\attack-btn.png"
```
Captures a single template, saves it to the specified path, then exits. BotBuilder watches for process exit and reloads the field automatically.

### 7.2 Capture Flow

**Step 1 — Pick a window**

BotCapture opens with a list of all visible windows (process name + window title + a small live thumbnail). The user selects the window they want to capture from. This scopes all subsequent captures to that window.

**Step 2 — Select a region**

BotCapture takes a screenshot of the selected window and displays it fullscreen (or in a large panel). The user drags a selection rectangle over the region they want to use as a template. The selected region is highlighted; everything outside is dimmed.

**Step 3 — Preview & Confirm**

A confirmation panel appears showing:

- **Template preview** — the cropped region at actual size and 2× zoom
- **Filename field** — pre-filled with a generated name (e.g. `capture_001.png`); user can rename
- **Confidence threshold slider** — range 0.0–1.0, default 0.9
- **Test Match button** — takes a fresh screenshot of the source window, runs an OpenCV template match, and overlays the result:
  - 🟢 Green highlight box = match found at or above the threshold
  - 🔴 Red = no match found at current threshold
  - Match score displayed numerically alongside the highlight
- The user can adjust the slider and re-test until the match behaviour looks right
- **Save** commits the file (and exits in integrated mode); **Retake** goes back to step 2

### 7.3 Confidence Threshold Persistence

When saved, the chosen confidence threshold is written to a sidecar file alongside the image:

```
attack-btn.png
attack-btn.png.meta.json   →  { "confidence": 0.88 }
```

When BotBuilder's image path field is populated (either via BotCapture or manual browse), it reads the `.meta.json` sidecar if present and pre-fills the confidence field in the Properties Panel. Users who browse to an image without a sidecar still set confidence manually.

### 7.4 Standalone Session Management

In standalone mode, BotCapture shows a simple session panel:

```
┌─────────────────────────────────────────┐
│  BotCapture                             │
├─────────────────────────────────────────┤
│  Source Window: [BlueStacks        ▾ ]  │
│  Save Folder:   [C:\Bots\assets\   📁]  │
├─────────────────────────────────────────┤
│  attack-btn.png          0.88   🟢  🗑  │
│  health-bar.png          0.90   🟢  🗑  │
│  victory-screen.png      0.75   🔴  🗑  │
├─────────────────────────────────────────┤
│  [ + New Capture ]                      │
└─────────────────────────────────────────┘
```

The session list shows all captures from the current session with their saved confidence and a quick re-test indicator. Clicking a row re-opens the preview/confirm panel for that template. The 🔴 indicator on `victory-screen.png` signals the threshold may need adjustment — the user can click in and fix it without recapturing.

## 8. Key Technical Decisions

| Decision | Choice | Rationale |
|---|---|---|
| UI framework | WPF | Native Windows, mature, well-suited for custom canvas rendering |
| Canvas rendering | Custom WPF `Canvas` + `DrawingVisual` | Performant node graph; no suitable off-the-shelf WPF graph library |
| Persistence | `.bot` (JSON) file per bot | Portable, human-readable, easy to back up or version control |
| Async model | `Task`-based with `CancellationToken` | Keeps runner responsive; cancellation is first-class |
| Extensibility | `IActionDefinition` / `IActionExecutor` | New action types added without touching the core engine |
| DI container | `Microsoft.Extensions.DependencyInjection` | Lightweight, works well in both WPF and console contexts |
| Variable system | Simple key/value (`Dictionary<string, object>`) | Sufficient for most flows; complex types handled via Lua scripting action |
| Graph topology | DAG only (no cycles) | Loops via Loop action; prevents infinite loop bugs by construction |
| Screen automation | OpenCvSharp (template matching) | Games don't expose accessibility trees; image recognition is the right primitive |
| Android automation | SharpAdbClient + ADB | Communicates with ADB daemon over TCP; no process-spawning overhead |
| Browser automation | Playwright (C# bindings) | Best-in-class, cross-browser, well-maintained |
| Desktop UI automation | FlaUI (UIA3 wrapper) | Clean .NET API over Windows UI Automation |
| Scripting | NLua (Lua) | Lightweight, embeddable, no external runtime; gentle learning curve |
| Error handling | Halt by default; per-action retry policy | Predictable failure behaviour; retry is essential for flaky image recognition |
| Parallel error strategy | Configurable enum on Run Parallel action | HaltAll / WaitThenHalt / Continue — user chooses the right tradeoff per bot |
| Target resolution | Named targets in `.bot`; selectors passed via CLI at runtime | Decouples bot definition from machine-specific handles; scriptable and automatable |
| Target picker (editor) | Live UI dialog showing available processes/devices; displays equivalent CLI command | Approachable for new users; teaches CLI naturally rather than hiding it |
| Multi-target support | Multiple named `BotTarget`s per bot; `TargetId` per action | Enables multi-client botting; defaults to first target for simple single-target bots |
| BotCapture IPC | `--output` path + process exit watch | Simple, no messaging protocol needed; consistent with how BotBuilder launches BotRunner |
| Confidence UX | Live OpenCV test match with visual highlight + slider | Removes guesswork; users tune threshold against real window content |
| Confidence persistence | `.meta.json` sidecar alongside template image | Keeps metadata portable with the image; no database needed |
